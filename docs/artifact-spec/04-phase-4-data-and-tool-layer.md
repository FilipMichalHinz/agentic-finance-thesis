**PHASE 4 — DATA AND TOOL LAYER**

1. **Data-layer overview**

The artifact should use a controlled data-layer architecture in which external APIs are used only for data ingestion, while Supabase serves as the runtime data source for the system during backtesting.

The external ingestion sources are:

* yfinance for daily market price data  
* EDGAR for SEC filing metadata and filing events  
* FMP for fundamental ratios, stock-specific news, general news, and macroeconomic indicators

The operational data source for the multi-agent system is Supabase. During evaluation runs, the agents should not query live APIs directly. They should consume stored data from Supabase in order to preserve reproducibility, comparability, and control.

2. **Source tables and prepared tables**

The data layer should distinguish between:

* source-oriented tables  
* prepared package tables

The source-oriented tables are the canonical stored layer and contain the ingested historical data from the upstream sources.

These source-oriented tables include, at minimum:

* market price data  
* technical indicator data  
* fundamental ratio data  
* stock-specific news data  
* general news data  
* economic indicator data  
* filing metadata and filing-linked knowledge content

In addition to the source-oriented tables, the system should use prepared package tables in Supabase. These tables should be generated by scripts and should contain the standardized daily screening inputs used by the agents during runtime.

This design is preferable to recomputing the full package at runtime because it improves:

* reproducibility  
* reuse across configurations  
* reuse across clean and disinformed dataset variants  
* runtime efficiency  
* auditability of what each agent saw on a given day


3. **Source-oriented tables**

The source-oriented layer should include the tables already collected in Supabase, or their cleaned equivalents.

The market price table should store daily OHLCV-style data, including:

* ticker  
* event timestamp  
* open  
* high  
* low  
* close  
* volume

The technical indicator table should store daily technical indicator values linked to a ticker and date, including:

* indicator name or column  
* timeframe  
* period length where relevant  
* event date  
* raw payload where needed for traceability

The fundamental ratio table should store normalized company-level financial and valuation indicators, including fields such as:

* liquidity ratios  
* margin ratios  
* return ratios  
* leverage ratios  
* turnover ratios  
* valuation multiples  
* dividend yield  
* source period metadata  
* reporting period metadata  
* raw payload where needed for traceability

The stock-news table should store stock-specific news rows with at least:

* ticker  
* title  
* content or summary text  
* published timestamp  
* publisher or source  
* deduplication key

The general-news table should store market-wide or economy-wide news with comparable metadata.

The economic-indicators table should store macroeconomic indicators such as:

* GDP  
* unemployment rate  
* CPI  
* inflation rate  
* event date  
* value

The filing-related layer should store:

* filing metadata from EDGAR  
* filing type  
* filing date  
* acceptance timestamp  
* accession number  
* source links  
* filing-linked knowledge-base content where available


4. **Prepared package tables**

The prepared package layer should be implemented as Supabase tables populated by scripts from the source-oriented tables.

These prepared tables should contain the daily screening inputs that are delivered to the agents during runtime.

The prepared package layer should be versionable so that the same package set can be reused across:

* all four configurations  
* clean dataset runs  
* disinformed dataset runs  
* different scenario IDs or simulation variants

The prepared package tables may be separated by role or merged into one structured daily package table, depending on implementation convenience. In either case, they should support fields such as:

* package date  
* ticker  
* dataset version  
* scenario ID  
* manipulation mode  
* package type or agent role


5. **Precomputed daily package fields**

The daily information package should include precomputed daily change fields rather than recomputing those changes during every runtime cycle.

This is important because the daily package is primarily a screening layer, and the screening logic depends on day-to-day changes and trigger-like signals.

The prepared package layer should therefore include, where appropriate:

* price change from previous close to current open  
* price change from open to close  
* close-to-close price change  
* price changes in nominal terms and/or percentage terms  
* day-to-day changes in technical indicator values  
* selected normalized change fields for fundamentals where a new value became available  
* event flags for filings and other point-in-time updates

For fundamentals, the package should not force artificial daily changes when no new report arrived. Instead, it should use:

* the latest available point-in-time value  
* change from the prior available value where relevant  
* event flags for filings or newly available financial information


6. **Daily package role mapping**

The Technical Analyst’s daily package should include:

* relevant price data  
* selected daily price-change fields  
* technical indicator values  
* day-to-day changes in those indicators

The News Analyst’s daily package should include:

* relevant price data  
* one selected stock-specific news item for the date  
* one summary of that selected news item  
* daily news count for the stock  
* selected market-wide or general news context

The Fundamental Analyst’s daily package should include:

* relevant price data  
* normalized fundamental ratios  
* changes in those values where a new observation became available  
* macroeconomic context  
* filing flags and related event indicators

The Portfolio Manager does not receive the same screening package as the specialist analysts. Instead, the Portfolio Manager works primarily with:

* analyst outputs  
* current portfolio state  
* candidate state  
* selected notes and summaries  
* challenge and debate outputs where applicable


7. **News-of-the-day selection rule**

For the News Analyst’s daily package, the system should use one selected stock-specific news item per stock per day.

The selection rule should be:

* the latest stock-specific article published on that date before the decision cutoff

This rule is simple, defendable, and easy to apply consistently across the backtest.

The package should also include the stock’s daily news count so the agent can see whether the selected item came from a more active or quieter news day.

The full historical news rows should remain available in the source-oriented tables for deeper analysis.

8. **Manipulated news design**

The system should maintain both:

* a clean stock-news dataset  
* a manipulated stock-news dataset

The manipulated stock-news dataset should be stored in a separate table in Supabase.

Each manipulated row should contain both the original and manipulated information, so that the system and the researcher can trace exactly what was changed.

The manipulated stock-news table should include fields such as:

* original news reference ID or deduplication key  
* ticker  
* original title  
* original content or summary  
* manipulated title  
* manipulated content or summary  
* fake-news boolean flag  
* severity label  
* scenario ID  
* activation date  
* creation timestamp

This design is useful for:

* traceability  
* auditability  
* attack-success coding  
* scenario control  
* later comparison between clean and manipulated conditions


9. **Disinformation dataset logic**

The system should support at least two manipulated-news events per stock where feasible:

* one stronger high-impact fake-news event  
* one milder fake-news event

The occurrence of these manipulated-news days may be randomized within the defined experimental logic, but the resulting dataset version should be frozen once created for a given experiment.

The manipulated dataset and the clean dataset should both remain stored in Supabase so that the same backtest period can be run under either condition.

10. **Filing and knowledge-base usage**

Any relevant filing present in the allowed forms should act as a trigger or flag for the Fundamental Analyst during screening.

The filing forms currently considered should include:

* 10-K  
* 10-Q  
* 8-K

During deeper analysis, the Fundamental Analyst should also be allowed to retrieve filing-linked content from the knowledge-base layer when relevant.

This means the filing logic should work in two levels:

* screening level: filing event or filing flag  
* deep-analysis level: retrieval of filing-linked vectorized or indexed content from the knowledge-base layer

This keeps filings highly relevant to the Fundamental Analyst without forcing the daily package itself to carry large filing texts.

11. **Runtime retrieval versus precomputed delivery**

The system should follow a hybrid data-access design.

The following should be precomputed:

* daily agent packages  
* selected daily screening inputs  
* selected change fields  
* selected normalized ratios  
* one-news-of-the-day selection per stock  
* clean and manipulated package-ready news variants

The following should be retrieved at runtime during deeper analysis:

* deeper historical rows from role-relevant source tables  
* filing-linked knowledge-base content where relevant  
* prior notes and summaries  
* current portfolio state  
* candidate state  
* role-specific deeper analysis history

This hybrid design keeps screening stable and reproducible while allowing selective deeper analysis at runtime.

12. **Role-specific data access**

Each agent should be allowed to access only the tables and columns relevant to its role.

The Technical Analyst should access:

* market prices  
* technical indicators  
* technical notes and summaries

The News Analyst should access:

* stock-specific news  
* general news  
* relevant price data  
* news notes and summaries  
* manipulated news when the scenario requires it

The Fundamental Analyst should access:

* normalized fundamental ratios  
* filing metadata  
* filing-linked knowledge-base content  
* economic indicators  
* relevant price data  
* fundamental notes and summaries

The Portfolio Manager should access:

* analyst outputs  
* candidate state  
* current portfolio state  
* selected recent summaries or notes  
* debate outputs where present  
* challenger outputs where present

This role-based data access should be enforced through the application or backend layer rather than by giving all agents unrestricted direct access to all tables.

13. **Point-in-time correctness**

The data layer must preserve point-in-time correctness.

This means the daily package generator and deeper analysis retrieval logic must use only data that would have been available by the decision cutoff for trading day t.

The relevant timestamps should be respected, including:

* event timestamps for price data  
* publication timestamps for news  
* filing date and acceptance timestamp for filings  
* event dates for economic indicators  
* reporting-period availability for fundamentals

This is essential for preventing future-information leakage in the backtest.

14. **Memory and continuity artifacts**

Continuity across trading days should be handled through structured records stored in Supabase.

These artifacts should include:

* analyst stock notes  
* weekly summaries  
* candidate-state records  
* portfolio decision records

The system should use a latest-snapshot-plus-history logic:

* one current structured record for quick retrieval  
* a historical trail of prior records for traceability

Memory retrieval should begin as structured database retrieval rather than full vector-RAG by default.

Vector or embedding-based retrieval may remain available in the filing-linked knowledge-base layer where it already exists, but the broader memory design should remain primarily structured and relational.

15. **Additional runtime tables**

In addition to the source-oriented and prepared package tables, the system should maintain a small set of persistent runtime tables for the artifact itself.

These should include, at minimum:

* candidate state  
* analyst notes  
* weekly summaries  
* portfolio state  
* portfolio history  
* run logs  
* evaluation event logs

These tables are necessary because they are part of the artifact’s continuity and auditability, not just the raw data ingestion layer.

16. **Supabase access model**

The Phase 4 design should assume backend-controlled access to Supabase.

This means:

* the backend or application service queries and writes the research data  
* the agents do not directly access wide-open database tables  
* the schema remains compatible with safer access control patterns

This is sufficient for the thesis artifact and avoids unnecessary security overengineering.

17. **Phase 4 summary**

The data and tool layer should use yfinance, EDGAR, and FMP as ingestion sources, while Supabase serves as the actual runtime data source for the system during backtesting. The data model should distinguish between source-oriented tables and precomputed prepared package tables. The daily information packages should be generated in advance and stored in Supabase so that all configurations consume the same screening inputs. These packages should include daily nominal or percentage changes for prices and technical indicators, along with point-in-time change logic or event flags for fundamentals and filings. Clean and manipulated stock-news datasets should both be stored in Supabase, with manipulated rows linked back to their original source records. Deeper analysis should retrieve role-specific historical rows, notes, summaries, candidate state, portfolio state, and filing-linked knowledge-base content at runtime. The overall design should prioritize reproducibility, point-in-time correctness, and controlled role-based access.
